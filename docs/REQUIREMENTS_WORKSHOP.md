# 요구사항 워크숍 가이드

## 목적

이 문서는 JSON 기반 데모를 관련자에게 보여준 뒤, 실제 웹사이트/CMS 프로젝트 요구사항을 확정하기 위한 질문지입니다.

데모 미팅의 목표는 "좋다/나쁘다" 의견을 듣는 것이 아니라, 실제 프로젝트 범위에 영향을 주는 결정을 내리는 것입니다.

## 참석자

권장 참석자:

- 마케팅 담당자
- 기획 담당자
- 브랜드/디자인 담당자
- 개발 담당자
- 인프라/보안 담당자
- 최종 승인권자 또는 의사결정 대리자

## 미팅 전 준비

- 데모 페이지 접속 URL
- `/admin-demo` 접속 URL
- `ref-image` 4개 기준 화면
- 이 문서의 질문 목록
- 결정사항 기록 양식

## 데모 시나리오

1. 공개 메인 페이지를 먼저 보여준다.
2. 제품 탭을 전환해 D.AMO, WAPPLES, iSIGN, Cloudbric 콘텐츠 변경을 보여준다.
3. 관리자 데모에서 Hero 텍스트를 수정한다.
4. 뉴스 항목을 추가하거나 순서를 바꾼다.
5. 구독 영역에서 문구, 입력 필드, 버튼 순서를 바꾼다.
6. 수상 항목을 수정한다.
7. 섹션 순서를 바꿔 공개 페이지 반영 흐름을 보여준다.
8. 실제 프로젝트에서 필요한 추가 기능을 질문한다.

## 핵심 결정 질문

### 1. 편집 범위

- 마케터가 직접 수정해야 하는 항목은 어디까지인가?
- 텍스트, 이미지, 링크 외에 색상이나 간격도 수정해야 하는가?
- 섹션 순서 변경만 필요한가, 섹션 추가/삭제도 필요한가?
- 같은 섹션을 여러 번 사용할 수 있어야 하는가?
- 섹션 내부 요소의 순서 변경이 필요한 곳은 구독 영역 외에도 있는가?

권장 기본값:

- 텍스트, 이미지, 링크, 반복 항목, 섹션 순서까지 허용
- 색상, 간격, 픽셀 단위 위치 조정은 허용하지 않음

### 2. 페이지 범위

- 1차 오픈에 필요한 페이지는 무엇인가?
- 메인 페이지 외 제품 상세 페이지가 필요한가?
- 회사 소개, 채용, 문의, 자료 다운로드, 블로그/뉴스 상세 페이지가 필요한가?
- 기존 워드프레스 콘텐츠 중 반드시 이관해야 하는 페이지는 무엇인가?

결정 필요:

- 1차 오픈 필수 페이지
- 2차 이후 확장 페이지
- 폐기할 기존 콘텐츠

### 3. 뉴스/보도자료

- 뉴스는 단순 목록인가, 상세 페이지가 필요한가?
- 카테고리나 태그가 필요한가?
- NEW 배지는 수동 설정인가, 게시일 기준 자동인가?
- 게시 예약이 필요한가?
- 목록 노출 개수는 관리자가 정해야 하는가?

권장 기본값:

- 실제 프로젝트에서는 `News` Collection으로 분리
- 제목, 요약, 본문, 대표 이미지, 날짜, 카테고리, 노출 여부 관리

### 4. 제품 콘텐츠

- 제품 탭은 현재 4개가 고정인가?
- 제품이 추가될 가능성이 있는가?
- 제품별 상세 페이지가 필요한가?
- 제품 로고와 시각 에셋은 마케터가 교체해야 하는가?
- 제품 카테고리 체계는 현재와 같이 데이터/웹/인증/클라우드 보안으로 유지되는가?

권장 기본값:

- 데모에서는 4개 탭 고정
- 실제 프로젝트에서는 제품 확장 가능성을 고려해 Collection 전환 여부 검토

### 5. 구독 영역

- 구독 버튼은 어디로 연결되어야 하는가?
- 실제 이메일 수집이 필요한가?
- 개인정보 동의 체크박스가 필요한가?
- 수집한 이메일은 어디에 저장해야 하는가?
- 사내 이메일/CRM/마케팅 툴과 연동해야 하는가?

권장 기본값:

- 데모에서는 UI만 제공
- 실제 연동은 별도 보안/개인정보 검토 후 결정

### 6. 이미지와 파일 (Asset Manager)

- 마케터가 직접 업로드해야 하는 파일 유형은 무엇인가?
- 이미지(배경 이미지 포함), 동영상(mp4), PDF, 폰트, HTML 중 어떤 유형이 필요한가?
- 파일은 경로 입력이 아니라 로컬 파일 선택 업로드 방식이어야 하는가?
- 업로드한 모든 파일을 한 곳에서 관리·재사용하는 중앙 Asset Manager가 필요한가?
- 업로드 파일 크기 제한과 허용 확장자/MIME는 어떻게 할 것인가?
- alt 텍스트 입력을 필수로 할 것인가?
- 폰트/HTML처럼 디자인·보안에 영향이 큰 파일은 누가 어디까지 등록할 수 있는가?
- 기존 이미지 자산은 어디에서 관리하고 어떻게 이관할 것인가?

권장 기본값:

- 미디어는 로컬 업로드 방식으로 전환하고, 중앙 Asset Manager(Payload `Media`(Upload) 컬렉션 + MinIO 기반)에서 관리
- 콘텐츠 필드는 경로 문자열 대신 Media 참조로 전환하되, 컴포넌트 props(URL 문자열)는 유지
- 1차 허용: 이미지와 PDF. 동영상(mp4)은 스트리밍/용량 정책 확정 후 허용
- 폰트는 임의 업로드가 아니라 등록된 폰트 토큰으로 통제, HTML은 sanitize·격리·접근 제한 등 통제된 방식으로만 허용
- 이미지 alt 텍스트 필수, 업로드 확장자/MIME/용량 제한과 공개·비공개 버킷 분리 적용

세부 전환 방향은 [CMS_CONTENT_MODEL.md](./CMS_CONTENT_MODEL.md) "Asset Manager (미디어 관리) 전환 방향", 운영/보안 기준은 [LOCAL_AND_DOCKER.md](./LOCAL_AND_DOCKER.md)를 참고합니다.

### 7. 다국어

- 한국어만 필요한가?
- 영문 사이트가 필요한가?
- 언어별 콘텐츠를 동일 구조로 관리해야 하는가?
- 언어별 발행 시점이 다를 수 있는가?

권장 기본값:

- 실제 확장 가능성이 있으면 처음부터 다국어 모델을 고려
- 데모에서는 한국어만 구현

### 8. 승인/발행 프로세스

- 마케터가 수정하면 바로 공개되어도 되는가?
- 검수자 승인 후 발행해야 하는가?
- 초안/미리보기가 필요한가?
- 예약 발행이 필요한가?
- 이전 버전으로 되돌리기가 필요한가?

권장 기본값:

- 실제 프로젝트에서는 Draft/Preview는 필수에 가깝게 고려
- 승인 워크플로우는 조직 운영 방식에 따라 결정

### 9. 관리자 권한

- 관리자 역할은 몇 단계가 필요한가?
- 마케팅 담당자가 모든 페이지를 수정해도 되는가?
- 제품 담당자는 제품 섹션만 수정해야 하는가?
- 외부 협력사 계정이 필요한가?
- 관리자 활동 로그가 필요한가?

권장 기본값:

- Admin
- Editor
- Viewer
- 필요 시 Product Editor

### 10. 운영/보안

- 관리자 페이지는 내부망에서만 접근하는가?
- VPN 또는 IP 제한이 필요한가?
- 사내 SSO 연동이 필요한가?
- 백업 보관 기간은 어떻게 할 것인가?
- 장애 발생 시 누가 복구를 담당하는가?

권장 기본값:

- `/admin` 내부망/IP 제한
- DB와 Storage 정기 백업
- 최소 1회 복구 리허설

## 결정 기록 양식

미팅 중 아래 형식으로 결정사항을 기록합니다.

```md
## 결정사항

- 결정: 섹션 추가/삭제는 1차 오픈에서 제외하고 순서 변경만 허용한다.
- 이유: 디자인 안정성과 데모 범위를 유지하기 위해.
- 영향: CMS 스키마는 고정 블록 + reorder 중심으로 설계한다.
- 후속 작업: 2차에서 섹션 추가 기능 필요 여부 재검토.
```

## 우선순위 분류

미팅 결과는 아래 기준으로 분류합니다.

### Must

1차 오픈에 반드시 필요합니다. 없으면 운영이 어렵습니다.

예:

- 메인 페이지 텍스트/링크 수정
- 뉴스 등록
- Footer 메뉴 수정
- 미리보기

### Should

중요하지만 1차 오픈 이후 보강할 수 있습니다.

예:

- 예약 발행
- 버전 되돌리기
- 다국어

### Could

있으면 좋지만 프로젝트 리스크를 키울 수 있습니다.

예:

- 인컨텍스트 비주얼 편집
- 복잡한 드래그 앤 드롭 빌더
- 고급 승인 워크플로우

### Won't for v1

1차 버전에서는 제외합니다.

예:

- 픽셀 단위 자유 배치
- Webflow/Elementor 수준의 전체 페이지 빌더
- 복잡한 마케팅 자동화
